# CMP Agent — Monitoring & Metrics Collection

## Overview

The CMP Agent is a lightweight monitoring agent installed on provisioned VMs. It collects real-time system metrics (CPU, memory, disk, network, OS info, top processes) and pushes them to the CMP platform via authenticated HTTPS POST at regular intervals.

This enables CMP to provide live resource health dashboards, historical performance trends, and proactive alerting — all without requiring external monitoring tools.

---

## Key Capabilities

| Capability | Description |
|-----------|-------------|
| System Metrics | CPU usage, memory utilization, disk volumes, network interfaces |
| OS Information | OS type, distro, kernel version, hostname, architecture, uptime |
| Process Monitoring | Top processes by CPU/memory (up to 20) |
| Custom Metrics | Extensible key-value metrics for application-specific data |
| Secure Communication | Agent authenticates with a unique API key issued during registration |
| Auto-Expiry | Metrics records include TTL for automatic DynamoDB cleanup |

---

## How It Works

### 1. Agent Registration

When a VM is provisioned through CMP, a one-time **registration token** is automatically generated by the task runner and injected into the task execution context. This allows provisioning tasks (e.g., Terraform, cloud-init scripts) to embed the token in the VM's startup sequence without manual intervention.

1. The task runner generates a registration token tied to the resource and execution
2. The provisioning task embeds the token in the VM's user-data or startup script
3. Agent starts on the VM and calls the CMP registration endpoint with the one-time token
4. CMP validates the token, creates an agent record, and returns a long-lived **agent API key**
5. The agent stores the API key locally for ongoing metric reporting

The token, agent endpoint, install script URL, and recommended report interval are all available in the task context under `context.agent`, enabling zero-touch agent deployment.

### 2. Metrics Reporting

Once registered, the agent operates in a push model:

- Collects system metrics every reporting interval (default: 60 seconds)
- Pushes a metrics payload to the CMP metrics endpoint using the agent API key
- CMP stores the metrics in DynamoDB with a TTL for automatic expiry

### 3. Health Status

CMP tracks agent health based on the `last_seen_at` timestamp:

- **Connected**: Last metric received within 2 reporting intervals
- **Disconnected**: No metrics received beyond 2 reporting intervals
- **Inactive/Revoked**: Agent explicitly disabled or token revoked

---

## Agent Statuses

| Status | Meaning |
|--------|---------|
| `active` | Agent is registered and expected to report metrics |
| `inactive` | Agent is paused or the associated resource is stopped |
| `revoked` | Agent token has been revoked — agent cannot report metrics |

## User-Facing Features

### Resource Metrics Dashboard (UI)

The **ResourceAgentMetrics** component renders directly on the resource detail page, providing a comprehensive system health view without leaving CMP.

#### Dashboard Sections

| Section | What's Shown |
|---------|-------------|
| **Agent Status Header** | Connection status (connected/disconnected), hostname, agent version, IP address, last-seen timestamp, manual refresh button |
| **Operating System** | OS/distro, kernel version, architecture, system uptime |
| **CPU** | Current usage percentage with color-coded progress bar, core count, load averages (1m/5m/15m) |
| **Memory** | Usage percentage with progress bar, used/total breakdown, available memory, swap usage |
| **Disk Volumes** | Table of all mounted volumes — device, mount point, filesystem, total/used/free space, usage bar |
| **Network Interfaces** | Cards per interface showing name, IP, MAC, TX/RX bytes, up/down indicator |
| **Top Processes** | Table of top 10 processes by resource consumption — PID, name, CPU%, memory%, status |
| **Recent History** | Last 15 data points showing CPU%, memory%, and disk% over time |

#### Auto-Refresh

The dashboard auto-refreshes every 60 seconds. Users can also trigger an immediate refresh via the refresh button in the status header.

#### No Agent Installed State

If no agent is detected for the resource, the UI displays an installation guide with a Python provisioning task example showing how to embed agent installation in a user's provisioning task:

```python
# In your provisioning task (Python):
agent = cmp["agent"]

user_data = f"""#!/bin/bash
# Your setup commands here...
yum install -y nginx

# Install CMP Agent
curl -sSL {agent['install_url']} | bash -s -- \\
  --endpoint {agent['endpoint']} \\
  --token {agent['token']} \\
  --resource-id {instance_id} \\
  --tenant-id {cmp['execution']['tenant_id']}
"""

# Pass user_data to your VM creation call (run_instances, custom_data, etc.)
```

This snippet is intended for use within a CMP provisioning task, where all `cmp` context variables (agent endpoint, token, tenant ID) and the `instance_id` are available automatically at runtime.

> **Note:** The install script must be run as root. If invoked as a non-root user, it exits with a clear error and prints the exact `sudo` command to use. Always prepend `sudo` when piping to `bash` on non-root shells.

### Install Agent Action (Post-Provisioning)

For VMs that are already running (not provisioned with the agent), users can install the agent directly from the **Resource Detail → Actions** section.

#### How It Works

1. Navigate to a running VM's resource detail page
2. Click the **Install Agent** button (purple button, visible only when status = `running`)
3. CMP first attempts **automatic installation** via cloud provider APIs:

| Provider | Mechanism | Requirement |
|----------|-----------|-------------|
| AWS | SSM SendCommand (`AWS-RunShellScript`) | SSM Agent running + IAM instance profile with `AmazonSSMManagedInstanceCore` |
| Azure | VM Run Command | Azure VM Agent running + `runCommand` permission on the service principal |
| GCP | Metadata startup-script | `compute.instances.setMetadata` permission on the service account |

4. If auto-install succeeds → success alert shown, agent starts reporting in 2-3 minutes
5. If auto-install fails → modal appears with the manual install command for the user to copy and run via SSH

---

#### IAM Role & Permissions Required (AWS)

For the **auto-install via SSM** to work, the EC2 instance must have an IAM instance profile with the `AmazonSSMManagedInstanceCore` managed policy. This allows the SSM Agent on the instance to communicate with AWS Systems Manager.

**Setup steps:**

1. **Create an IAM Role** (one-time setup):
   - Trusted entity: `EC2`
   - Attach managed policy: `arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore`
   - Suggested name: `CMP-EC2-SSM-Role`

2. **Create an Instance Profile** and add the role:
   ```bash
   aws iam create-instance-profile --instance-profile-name CMP-EC2-SSM-Profile
   aws iam add-role-to-instance-profile --instance-profile-name CMP-EC2-SSM-Profile --role-name CMP-EC2-SSM-Role
   ```

3. **Attach to instances** (one of):
   - At launch: include `IamInstanceProfile` in `run_instances`
   - Post-launch: `aws ec2 associate-iam-instance-profile --instance-id <id> --iam-instance-profile Name=CMP-EC2-SSM-Profile`

4. **Wait 2-3 minutes** for the SSM Agent to register with Systems Manager

**Minimum permissions on the CMP credential** (the IAM user/role that CMP uses to call SSM):
- `ssm:SendCommand`
- `ssm:DescribeInstanceInformation`

**Note:** Most modern AMIs (Amazon Linux 2, Amazon Linux 2023, Ubuntu 20.04+, Windows Server 2016+) come with SSM Agent pre-installed. If the instance uses a custom AMI without SSM Agent, install it manually: `sudo yum install -y amazon-ssm-agent && sudo systemctl enable --now amazon-ssm-agent`

---

#### Permissions Required (Azure)

For auto-install via Azure Run Command:

- **On the VM**: Azure VM Agent must be installed and running (default on all Azure Marketplace images)
- **On the CMP service principal**: `Microsoft.Compute/virtualMachines/runCommand/action` permission (included in the `Virtual Machine Contributor` role)

---

#### Permissions Required (GCP)

For auto-install via metadata startup-script:

- **On the CMP service account**: `compute.instances.setMetadata` permission (included in the `Compute Instance Admin` role)
- **On the instance**: Must be running and reachable from the GCP metadata API (default for all GCP instances)

---

#### Auto-Install Failure Reasons

| Reason | Explanation |
|--------|-------------|
| SSM Agent not installed/online | Instance doesn't have AWS Systems Manager agent running or has no IAM instance profile |
| No IAM instance profile | Instance has no role attached — SSM Agent can't authenticate with AWS |
| Insufficient IAM permissions | CMP credential lacks `ssm:SendCommand` permission |
| Azure VM Agent not responding | VM Agent is unhealthy or VM is not fully booted |
| GCP metadata permission denied | Service account lacks `compute.instances.setMetadata` |
| SDK not available | Required cloud SDK not installed on CMP backend |

In all failure cases, the user gets a fallback modal with the full install command they can run manually via SSH.

#### Install Agent API Response

When the `install_agent` action is triggered via `POST /api/v1/resources/{resource_id}/actions`, the `ActionResponse` includes agent-specific fields:

| Field | Type | Description |
|-------|------|-------------|
| `install_command` | `string \| null` | Fallback manual install command (shown when auto-install fails) |
| `auto_install_success` | `bool \| null` | Whether the automatic installation attempt succeeded |
| `auto_install_error` | `string \| null` | Error message if auto-install failed (e.g., "SSM Agent not online") |
| `registration_token` | `string \| null` | One-time token for agent registration (valid ~1 hour) |
| `agent_endpoint` | `string \| null` | CMP agent API base URL for the agent to connect to |
| `agent_status` | `string \| null` | Current agent status after install attempt (`active`, `pending`, etc.) |
| `agent_id` | `string \| null` | Assigned agent ID if registration completed immediately |
| `command_id` | `string \| null` | Cloud provider command ID for tracking async install (e.g., SSM command ID) |
| `expires_in_seconds` | `int \| null` | Seconds until the registration token expires |

Example response (auto-install failed, fallback provided):

```json
{
  "success": true,
  "message": "Auto-install failed. Use the manual install command below.",
  "auto_install_success": false,
  "auto_install_error": "SSM Agent not online on instance i-0abc123",
  "install_command": "curl -sSL https://cmp.example.com/agent/install.sh | bash -s -- --endpoint https://cmp.example.com/api/v1/agent --token cmp-reg-abc123 --resource-id i-0abc123 --tenant-id default",
  "registration_token": "cmp-reg-abc123",
  "agent_endpoint": "https://cmp.example.com/api/v1/agent",
  "expires_in_seconds": 3600
}
```

## Security Model

- **Automatic token generation**: Registration tokens are generated by the task runner during VM provisioning and scoped to the specific resource and execution
- **One-time registration tokens**: Single-use only; consumed upon successful agent registration
- **Agent API keys**: Long-lived but revocable; hashed (bcrypt) before storage
- **HTTPS only**: All agent-to-CMP communication encrypted in transit
- **Tenant isolation**: Agent records and metrics are scoped to the provisioning tenant
- **Token revocation**: Admins can revoke agent tokens to immediately block metric reporting
- **Graceful degradation**: If token generation fails, the task continues without agent info — no provisioning failure
- **Root requirement**: The install script requires root privileges. If run as a non-root user, it exits with a clear error message and provides the exact commands to re-run with `sudo` or as root. It does not attempt automatic elevation — this avoids silent privilege escalation and makes permission issues immediately visible

## Provisioning Task Examples

### AWS EC2 — Python Task

```python
import json
import boto3

# CMP injects cmp["agent"] automatically
agent = cmp.get("agent", {})

# Build user_data with agent install + instance ID resolution from metadata
user_data = f"""#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd

# Resolve instance ID from EC2 metadata (IMDSv2)
IMDS_TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 60")
INSTANCE_ID=$(curl -s -H "X-aws-ec2-metadata-token: $IMDS_TOKEN" http://169.254.169.254/latest/meta-data/instance-id)

# Install CMP Agent
curl -sSL {agent.get('install_url', '')} | bash -s -- \\
  --endpoint {agent.get('endpoint', '')} \\
  --token {agent.get('token', '')} \\
  --resource-id $INSTANCE_ID \\
  --tenant-id {cmp['execution'].get('tenant_id', 'default')}
"""

ec2 = boto3.client("ec2", region_name=cmp["credential"]["region"],
    aws_access_key_id=cmp["credential"]["aws_access_key_id"],
    aws_secret_access_key=cmp["credential"]["aws_secret_access_key"],
    aws_session_token=cmp["credential"]["aws_session_token"])

response = ec2.run_instances(
    ImageId=params["ami_id"],
    InstanceType=params.get("instance_type", "t3.micro"),
    MinCount=1, MaxCount=1,
    UserData=user_data,
    TagSpecifications=[{"ResourceType": "instance", "Tags": [
        {"Key": "Name", "Value": params.get("instance_name", "cmp-vm")}
    ]}]
)

instance_id = response["Instances"][0]["InstanceId"]
print(json.dumps({"instance_id": instance_id, "resource_id": instance_id}))
```

### AWS EC2 — Terraform Template

For Terraform-based provisioning, use a **two-step workflow**:

**Workflow DAG:**
```
[generate_agent_token] → [Terraform Apply — AWS EC2]
```

**Step 1: `generate_agent_token`** (Python task):
```python
import json

agent = cmp.get("agent", {})
output = {
    "cmp_agent_install_url": agent.get("install_url", ""),
    "cmp_agent_endpoint": agent.get("endpoint", ""),
    "cmp_agent_token": agent.get("token", ""),
    "cmp_agent_tenant_id": cmp.get("execution", {}).get("tenant_id", "default"),
    "status": "success",
}
print(json.dumps(output))
```

**Step 2: Terraform Apply** — map the step outputs as inputs in the DAG editor:

| Terraform Variable | Workflow Input Value |
|---|---|
| `instance_name` | `{{form.instance_name}}` |
| `ami_id` | `{{form.ami_id}}` |
| `region` | `{{form.region}}` |
| `subnet_id` | `{{form.subnet_id}}` |
| `cmp_agent_install_url` | `{{steps.generate_agent_token.cmp_agent_install_url}}` |
| `cmp_agent_endpoint` | `{{steps.generate_agent_token.cmp_agent_endpoint}}` |
| `cmp_agent_token` | `{{steps.generate_agent_token.cmp_agent_token}}` |
| `cmp_agent_tenant_id` | `{{steps.generate_agent_token.cmp_agent_tenant_id}}` |

> **Important:** Use **underscores** in step references (e.g., `generate_agent_token` not `generate-agent-token`). CMP normalises step IDs to underscores internally.

**Terraform Template** (`main.tf`):
```hcl
# CMP Agent variables (auto-injected via workflow step outputs)
variable "cmp_agent_install_url" {
  type    = string
  default = ""
}
variable "cmp_agent_endpoint" {
  type    = string
  default = ""
}
variable "cmp_agent_token" {
  type      = string
  default   = ""
  sensitive = true
}
variable "cmp_agent_tenant_id" {
  type    = string
  default = "default"
}

locals {
  install_agent = var.cmp_agent_token != "" && var.cmp_agent_install_url != ""

  agent_user_data = local.install_agent ? join("\n", [
    "#!/bin/bash",
    "sleep 10",
    "IMDS_TOKEN=$(curl -s -X PUT \"http://169.254.169.254/latest/api/token\" -H \"X-aws-ec2-metadata-token-ttl-seconds: 60\")",
    "INSTANCE_ID=$(curl -s -H \"X-aws-ec2-metadata-token: $IMDS_TOKEN\" http://169.254.169.254/latest/meta-data/instance-id)",
    "curl -sSL ${var.cmp_agent_install_url} | bash -s -- --endpoint ${var.cmp_agent_endpoint} --token ${var.cmp_agent_token} --resource-id $INSTANCE_ID --tenant-id ${var.cmp_agent_tenant_id}",
  ]) : ""
}

resource "aws_instance" "vm" {
  # ... other config ...
  user_data                   = local.agent_user_data != "" ? local.agent_user_data : null
  associate_public_ip_address = true  # Required for agent to reach CMP
}
```

> **Network requirement:** The instance must have outbound HTTPS access to CMP (public IP + IGW, or NAT Gateway).

### Azure VM (Python Task)

```python
import json, base64
from azure.identity import ClientSecretCredential
from azure.mgmt.compute import ComputeManagementClient

agent = cmp.get("agent", {})
cred = cmp["credential"]

cloud_init = f"""#!/bin/bash
apt-get update -y && apt-get install -y nginx
curl -sSL {agent.get('install_url', '')} | bash -s -- \\
  --endpoint {agent.get('endpoint', '')} \\
  --token {agent.get('token', '')} \\
  --resource-id {params['vm_name']} \\
  --tenant-id {cmp['execution'].get('tenant_id', 'default')}
"""

credential = ClientSecretCredential(cred["azure_tenant_id"], cred["azure_client_id"], cred["azure_client_secret"])
compute = ComputeManagementClient(credential, cred["azure_subscription_id"])

poller = compute.virtual_machines.begin_create_or_update(
    params["resource_group"], params["vm_name"],
    {"location": params.get("location", "eastus"),
     "hardware_profile": {"vm_size": params.get("vm_size", "Standard_B2s")},
     "os_profile": {"computer_name": params["vm_name"],
                    "admin_username": "azureuser",
                    "custom_data": base64.b64encode(cloud_init.encode()).decode()},
     "storage_profile": {"image_reference": {"publisher": "Canonical",
         "offer": "0001-com-ubuntu-server-jammy", "sku": "22_04-lts-gen2", "version": "latest"}}}
)
vm = poller.result()
print(json.dumps({"instance_id": vm.vm_id, "resource_id": params["vm_name"]}))
```

### GCP Compute Engine (Python Task)

```python
import json
from google.oauth2 import service_account
from googleapiclient.discovery import build

agent = cmp.get("agent", {})
cred = cmp["credential"]

startup_script = f"""#!/bin/bash
apt-get update -y && apt-get install -y nginx
curl -sSL {agent.get('install_url', '')} | bash -s -- \\
  --endpoint {agent.get('endpoint', '')} \\
  --token {agent.get('token', '')} \\
  --resource-id {params['instance_name']} \\
  --tenant-id {cmp['execution'].get('tenant_id', 'default')}
"""

sa_info = json.loads(cred.get("secret_key", "{}"))
credentials = service_account.Credentials.from_service_account_info(sa_info,
    scopes=["https://www.googleapis.com/auth/compute"])
compute = build("compute", "v1", credentials=credentials)

config = {
    "name": params["instance_name"],
    "machineType": f"zones/{params['zone']}/machineTypes/{params.get('machine_type', 'e2-micro')}",
    "disks": [{"boot": True, "autoDelete": True,
        "initializeParams": {"sourceImage": "projects/ubuntu-os-cloud/global/images/family/ubuntu-2204-lts"}}],
    "networkInterfaces": [{"network": "global/networks/default", "accessConfigs": [{"type": "ONE_TO_ONE_NAT"}]}],
    "metadata": {"items": [{"key": "startup-script", "value": startup_script}]}
}

compute.instances().insert(project=cred["project_id"], zone=params["zone"], body=config).execute()
print(json.dumps({"instance_id": params["instance_name"], "resource_id": params["instance_name"]}))
```

## Task Context Reference

Every CMP provisioning task receives the `cmp["agent"]` context automatically:

```python
cmp["agent"] = {
    "token": "cmp-reg-...",           # One-time registration token (valid 1 hour)
    "endpoint": "https://...",         # CMP agent API base URL
    "install_url": "https://.../install.sh",  # Install script URL
    "resource_id": "i-...",            # Pre-filled resource ID (from form_data)
    "report_interval": 60              # Reporting interval in seconds
}
```

The token is auto-generated per execution — no admin intervention needed.
